# Кабинет мастера — дизайн-спец (v.2, фаза 1)

> Это первая (и на момент написания — единственная) фаза v.2 upgrade-трека.
> Вторая фаза — demo-бот с реалистичным seed для продаж — откладывается на
> конец, после того как кабинет мастера отгружен и протестирован.

## Контекст

Существующий бот — один процесс, два режима: админ и клиент. Мастер заведён
как сущность (`masters` в БД) и получает push-уведомления через
`utils/notifications.py::notify_master`, но **не имеет собственного входа в
бота**. Если мастер пишет `/start`, он попадает в клиентский флоу (выбор
категории → услуги → ...) — то есть пытается записаться сам на себя. Это
плохо работает на салоне, где мастер хочет утром увидеть «что у меня
сегодня» без скроллинга по push-уведомлениям.

Кабинет мастера закрывает этот пробел: read-only pull-интерфейс внутри того
же бота, с тремя кнопками, без FSM и без новых таблиц.

## Цели и не-цели

**Цели.**

- Мастер со своим привязанным `user_id` видит свои записи (сегодня / все
  будущие) и своё недельное расписание.
- Один процесс, один бот. Мастер не ставит отдельное приложение, не создаёт
  отдельный бот.
- Ноль миграций БД, ноль новых зависимостей, ноль новых таблиц.
- Не ломать существующие админский и клиентский флоу.

**Не-цели (откладывается на следующие фазы, если вообще).**

- Мастер не может менять своё расписание. Только админ.
- Мастер не может отменять или переносить записи.
- Мастер не видит чужих мастеров, финансов салона, настроек, клиент-карт.
- Мастер не создаёт блокировки слотов.
- Не добавляем тесты (по правилу проекта).
- Не меняем существующий `notify_master` — push-уведомления уже работают,
  кабинет это **pull**, они не дублируют функцию.

## Архитектура

### Новый файл

`handlers/master.py` — собственный `Router()` с двумя фильтрами:

```python
router.message.filter(IsMasterFilter())
router.callback_query.filter(IsMasterFilter())
```

Внутри — три обработчика reply-кнопок и callback-обработчики для встроенной
навигации (edit_text по панели, паттерн `edit_panel_with_callback` из
`utils/panel.py`).

### Фильтр и кеш

В `utils/admin.py` уже есть `_db_admins_cache` и `refresh_admins_cache()`.
Добавляем симметрично:

- `_db_masters_cache: set[int]` — множество `user_id` активных мастеров с
  привязанным `user_id`.
- `refresh_masters_cache()` — вызывается на старте бота (`bot.py::main`,
  рядом с `refresh_admins_cache`) и после каждого мутирующего действия над
  мастерами: `create_master`, `update_master` (только если меняется
  `user_id`), `toggle_master_active`, `delete_master`.
- `is_master(user_id: int) -> bool` — проверка по кешу.
- `IsMasterFilter` — aiogram-фильтр по той же логике.

### Routing в `cmd_start`

`handlers/client.py::cmd_start` сейчас:

```python
if message.from_user.id in ADMIN_IDS:
    # admin
else:
    # client flow (new / returning)
```

Становится:

```python
if message.from_user.id in ADMIN_IDS or _db_admins_cache:
    # admin (без изменений)
elif await is_master(message.from_user.id):
    # master cabinet entry (NEW)
else:
    # client flow (без изменений)
```

Порядок критичен: админ может одновременно быть в `masters` (например,
владелица салона сама принимает клиентов). В этом случае она остаётся
админом.

### Регистрация роутеров в `bot.py`

```
admin.router, admin_appointments, admin_clients, admin_services,
admin_stats, admin_status, admin_settings, admin_blocks, admin_manage,
admin_masters, admin_export,
master.router,                       ← NEW, между admin и client
reviews.router,
client_reminders.router,
client_history.router,
client.router,                       ← catch-all в самом конце
```

Почему между admin и client: клиентский роутер содержит `F.text == "/start"`
и catch-all `fallback_message`. Если мастерский попадёт позже — клиентский
поймает его сообщения первым.

## UX

Кабинет открывается reply-клавиатурой с тремя кнопками:

```
┌──────────────────┐
│   📋 Сегодня     │
├──────────────────┤
│   📅 Мои записи  │
├──────────────────┤
│   📆 Моё расписание │
└──────────────────┘
```

Все три экрана — edit_text на одном сообщении-панели (`panel_msg_id` в
`utils/panel.py`, те же хелперы `edit_panel` / `edit_panel_with_callback`,
что и у админа).

### Экран «📋 Сегодня»

Показывает записи мастера **на сегодняшнюю дату**: scheduled + completed +
no_show. Отменённые (cancelled) — скрыты, чтобы не засорять.

Формат:

```
📋 Сегодня, 20 апреля

🕐 10:00 — Гульнара
   💅 Маникюр + гель-лак (1ч 30м)
   📞 +998 90 123 45 67
   ⏺ scheduled

✅ 12:30 — Лола
   💅 Педикюр (1ч)
   📞 +998 90 555 44 33
```

Иконка слева от времени: 🕐 scheduled, ✅ completed, 🚫 no_show.

Кнопка внизу: `🔄 обновить` (повторный запрос к БД).

Пустой список: «На сегодня записей нет. Хорошего дня.»

### Экран «📅 Мои записи»

Scheduled на сегодня и вперёд, без completed/no_show (для будущих они и так
не могут быть). Группировка по дате, до **30** записей (как в
`get_client_appointments` LIMIT). При превышении — внизу «... и ещё N».

Формат:

```
📅 Твои ближайшие записи

—— 20 апреля, пн ——
🕐 10:00 — Гульнара · Маникюр + гель-лак · 📞 +998 ...
🕐 14:00 — Лола · Педикюр · 📞 +998 ...

—— 21 апреля, вт ——
🕐 11:00 — ...
```

Пустой список: «Пока ничего запланировано.»

### Экран «📆 Моё расписание»

Недельная сетка из `master_schedule`:

```
📆 Твоё расписание

Пн  09:00 – 19:00
Вт  09:00 – 19:00
Ср  выходной
Чт  09:00 – 19:00
Пт  09:00 – 19:00
Сб  10:00 – 17:00
Вс  выходной

Изменить расписание — через администратора.
```

Если `master_schedule` пуст (новый мастер без персонального графика) —
показываем общий `weekly_schedule` с пометкой «общий график салона».

## Решения по дефолту

1. **Телефон клиента виден мастеру.** Альтернатива — скрыть ради защиты
   данных. Выбрано «виден», потому что мастер должен мочь позвонить клиенту
   если тот опаздывает.
2. **«📋 Сегодня» включает прошедшие сегодня визиты** (completed, no_show),
   а не только upcoming. Мастер утром/днём смотрит картину целиком.

## Что потребуется в `db/`

Большинство запросов уже есть. Нужно добавить:

- `get_master_by_user_id(user_id: int) -> dict | None` — для загрузки
  кеша `_db_masters_cache` и для вывода имени мастера в заголовке кабинета.
  Возможно уже есть под другим именем — проверить в `db/masters.py`.
- `get_master_appointments(master_id: int, date_from: str, limit: int = 30) -> list[dict]`
  — scheduled записи мастера от даты, до лимита. Возвращает тот же формат
  что `get_appointments_by_date_full`.
- `get_master_appointments_for_date(master_id: int, date: str) -> list[dict]`
  — для экрана «📋 Сегодня», включая completed/no_show, исключая cancelled.

Возможно часть этого можно выразить через существующие функции с
параметром `master_id` — на этапе имплементации проверить.

## Edge cases

- **Мастер без `user_id`.** `is_master(user_id)` вернёт False (в кеше нет),
  мастер пойдёт в клиентский флоу. Ожидаемо: без привязки TG-id мастер не
  может пользоваться кабинетом. Решается админом (привязать через
  «Мастера → редактировать → user_id»).
- **Гонка: деактивация во время сессии.** Мастер открыл кабинет, админ в
  параллели выключил его. Следующий клик → `IsMasterFilter` False → callback
  проваливается в клиентский роутер. Приемлемо для v1.
- **Мастер = админ одновременно.** Ветка `if admin` срабатывает первой —
  остаётся админом. Это корректно: у владелицы приоритет админ-панели.
- **Deactivated master jets a reminder push.** `notify_master` отдельно не
  проверяет `is_active` — если деактивировали за минуту до push-уведомления,
  мастер всё равно может получить сообщение. Не меняем в этой фазе:
  уведомление — информационное, сам кабинет закрыт фильтром.

## Verification

Ручная проверка автора:

1. В админ-панели создать мастера с TG-id, отличным от своего собственного
   админского (например, второй тестовый аккаунт).
2. С этого второго аккаунта написать боту `/start`.
3. Ожидаемо: появляется reply-клавиатура с тремя кнопками мастера, **не**
   клиентский флоу «ручки/ножки».
4. Создать тестовую запись этому мастеру (из админ-панели или от клиентского
   аккаунта) на сегодня.
5. Из аккаунта мастера нажать «📋 Сегодня» → увидеть созданную запись с
   именем, телефоном, услугой, временем.
6. Нажать «📆 Моё расписание» → увидеть недельную сетку.
7. Деактивировать этого мастера из админ-панели. С аккаунта мастера
   написать `/start` → упасть в клиентский флоу (выбор категории).

## Don'ts (чтобы не драйфнуть по ходу)

- Не делать FSM. Это чтение.
- Не трогать `notify_master`. Push уже работает, кабинет — pull.
- Не переписывать `cmd_start` — добавить `elif`, ничего не перестраивать.
- Не вводить «мой график можно менять» — это v.3 если вообще.
- Не давать мастеру любую агрегацию по салону (выручка, топ-услуги).
- Не добавлять тесты по умолчанию.
- Не рефакторить `utils/admin.py` при добавлении masters-кеша — только
  скопировать паттерн.

## Объём работы (Phase 1)

- `handlers/master.py` — новый файл, ~300–400 строк.
- `utils/admin.py` — +30 строк (masters-кеш, refresh, is_master,
  IsMasterFilter).
- `db/masters.py` — +2–3 новые функции, возможно часть существует.
- `bot.py` — +2 строки (импорт + `include_router(master.router)` в нужной
  позиции), +1 строка на `refresh_masters_cache()`.
- `handlers/admin_masters.py` — +3–4 места вызова `refresh_masters_cache()`
  после мутаций.
- `handlers/client.py::cmd_start` — +1 `elif` ветка.

# Phase 2 — Admin редактирует per-master расписание

> Добавлено 2026-04-20 после фидбека автора. Мастер читает своё
> расписание (Phase 1), но **никто** не мог его редактировать — таблица
> `master_schedule` заполнялась один раз при создании мастера копированием
> глобального `weekly_schedule`, и дальше оставалась замороженной.
>
> Это ломается на первом салоне с 2+ мастерами с разным графиком: Лола
> Пн-Пт, Гульнара Вт-Сб. Сейчас обе получат одинаковый салонный график и
> ни админ, ни мастер не могут поправить.

## Цели

- Админ может менять расписание конкретного мастера: toggle выходной,
  редактировать час начала, редактировать час конца, для каждого weekday.
- Мастер в своём кабинете по-прежнему только читает. Никакого нового
  доступа мастеру не даём.
- Глобальный `weekly_schedule` в «Настройки → График работы» остаётся как
  есть: он сидит **новых** мастеров при создании, существующих не трогает.

## Не-цели

- Не делаем «синхронизировать все master_schedule с глобальным одной
  кнопкой» — это может случайно затереть кастомизацию.
- Не делаем копирование «скопировать график мастера X → Y» — редко нужно,
  можно добавить позже.
- Не меняем структуру таблицы `master_schedule` (PK уже master_id+weekday,
  колонки те же).
- Не делаем фиксацию прошлых записей при сужении часов — только
  предупреждение о конфликтах, как уже работает в глобальных настройках.

## Архитектура

Зеркало существующего паттерна `handlers/admin_settings.py`:

- **Новый файл `handlers/admin_master_schedule.py`** — весь UI
  редактирования per-master расписания. В `handlers/admin_masters.py` не
  добавляем: этот файл уже 350 строк, легче понимать в отдельном.
- **Новая функция в `db/masters.py`**:
  `update_master_weekday(master_id, weekday, work_start, work_end)` —
  upsert через `INSERT OR REPLACE`. Значения `work_start=None,
  work_end=None` означают «выходной».
- **Новые FSM-states в `states.py`**:
  `master_schedule_edit_start`, `master_schedule_edit_end`.
- **Новые клавиатуры в `keyboards/inline.py`**:
  - `master_weekly_schedule_keyboard(master_id, schedule)` — 7 кнопок по
    дням недели с индикатором (часы или «—»), плюс «🔙 К мастеру».
  - `master_weekday_detail_keyboard(master_id, weekday, is_day_off)` —
    кнопки «🔄 Toggle выходной», «🕐 Час начала», «🕕 Час конца»,
    «🔙 К расписанию».
- **Кнопка на карточке мастера**: в `master_card_keyboard` добавить
  «📆 Расписание» с callback `master_sched_<master_id>`.
- **Регистрация роутера в `bot.py`**: `admin_master_schedule.router` —
  после `admin_masters.router` и до `master.router`.

## Callback namespace

Чтобы не конфликтовать с существующими `sched_*` (глобальное расписание):
- `master_sched_<id>` — открыть расписание мастера
- `msched_day_<id>_<weekday>` — детализация weekday
- `msched_toggle_<id>_<weekday>` — toggle выходной
- `msched_edit_start_<id>_<weekday>` — начать FSM на редактирование часа начала
- `msched_edit_end_<id>_<weekday>` — FSM на час конца

Префикс `msched_` (не `master_sched_`) для дочерних callbacks — короче и
не конфликтует ни с чем в проекте.

## Конфликты с существующими записями

При сужении часов (например, Пн 9-19 → 10-18) могут существовать
scheduled-записи вне новых часов. Копируем паттерн из
`admin_settings.py::_count_schedule_conflicts` — только фильтруем
дополнительно `WHERE master_id = ?`. Показываем предупреждение
«⚠️ есть N записей вне новых часов, проверьте «Все записи»» но изменение
всё равно применяем — как в глобальных настройках.

## Verification (ручная)

1. В админке: Мастера → создать «Тест-Мастер» → открыть карточку →
   появилась кнопка «📆 Расписание» → жми.
2. Открылась недельная сетка со стандартными часами (из salon-wide
   weekly_schedule на момент создания мастера).
3. Жми на «Среда» → детали дня → «🔄 Toggle выходной» → вернулись в
   сетку, среда показывает «—».
4. Снова Среда → деталь → «🕐 Час начала» → ввод `10` → сохранилось
   10:00-19:00.
5. Создай запись на ту же среду на 9:30 (из клиента) → вернись в админке
   в расписание мастера → смени Ср на 11:00-19:00 → должно показать
   «⚠️ есть 1 запись вне новых часов».
6. В кабинете мастера (со второго аккаунта-мастера) жми «📆 Моё
   расписание» → видишь обновлённые часы (из master_schedule, не
   глобального).
